# ESM Import Support for Scripts

**Date:** 2026-04-15
**Status:** Approved

## Problem

Scripts executed via `node -e` and `.jsh` files currently only support CommonJS-style `require()` calls. Skill scripts want to use standard ESM `import` syntax (`import chalk from 'chalk'`, `import './helpers.js'`). The existing execution engine wraps code in an `AsyncFunction` body where static `import` statements are syntactically invalid.

## Design

### Detection & Routing

Pre-scan the entry script source for static `import` statements (regex, same pattern as existing `extractRequireSpecifiers`). Two execution paths:

- **ESM detected** — new path: real module execution via `import()` + preview SW + import maps
- **CJS only / neither** — existing path: AsyncFunction + requireShim (unchanged)

This is a fork at the top of `executeJsCode()` / the `node -e` handler. Zero changes to the existing CJS path.

### Import Map Setup

Before the `import()` call, dynamically build and inject an import map:

1. **Pre-scan** all `.js`/`.mjs` files in the skill folder for bare import specifiers.
2. **Classify** each specifier:
   - Node built-ins (`fs`, `process`, `buffer`, `path`, `node:*`) — map to synthetic shim URLs (`/preview/__shims/fs.js`, etc.)
   - Unavailable built-ins (`http`, `crypto`, etc.) — map to an error shim that throws with a helpful message
   - Everything else (npm packages) — map to `https://esm.sh/<package>`
3. **Inject** a `<script type="importmap">` into the page (CLI mode) or sandbox iframe (extension mode) before the `import()` call.

### Synthetic Shim Modules

The preview service worker synthesizes shim responses on the fly for known `/__shims/*` paths. No physical VFS files needed. When the SW sees a request for `/preview/__shims/fs.js`, it generates the JavaScript response in memory, exporting from `globalThis.__slicc_*`.

Example synthesized shim:

```js
// Generated by preview-sw for /__shims/fs.js
const shim = globalThis.__slicc_fs;
export const readFile = shim.readFile;
export const writeFile = shim.writeFile;
export const readDir = shim.readDir;
export const exists = shim.exists;
export default shim;
```

Shims needed:

- `fs` — readFile, writeFile, readDir, exists, stat, mkdir, rm, fetchToFile
- `process` — argv, env, cwd, exit, stdout, stderr
- `buffer` — re-export globalThis.Buffer
- `path` — delegated to esm.sh/path-browserify via import map (no shim needed)
- Unavailable built-ins (`http`, `crypto`, etc.) — single error shim that throws with hint

### Execution Flow (ESM Path)

1. Set up shims on `globalThis` — `globalThis.__slicc_fs = fsBridge`, `__slicc_process`, `__slicc_exec`
2. Monkey-patch `console` — replace `console.log/error/warn/info` with capturing versions for stdout/stderr collection
3. Rewrite import specifiers to absolute URLs in the source (see below)
4. Create a `Blob` from the rewritten source and execute via `await import(blobUrl)`
5. Collect results — stdout/stderr from captured console output
6. Restore console, revoke blob URL
7. Clean up `globalThis.__slicc_*` properties

### Specifier Rewriting (not import maps)

**Why not import maps:** The original design used browser import maps (`<script type="importmap">`). This was abandoned because:

- The preview SW is registered with `scope: '/preview/'` so it only controls pages under `/preview/`
- `import()` calls from the main page at `/` are outside the SW scope — the SW never intercepts them
- Widening scope to `/` would cause projectRoot serve mode to intercept requests from the main page, breaking Vite HMR and app assets

**The actual approach — source rewriting:** Before executing, all import specifiers in the source are rewritten to absolute URLs:

- `import { readFile } from 'fs'` → `import { readFile } from 'http://localhost:5710/preview/__shims/fs.js'`
- `import chalk from 'chalk'` → `import chalk from 'https://esm.sh/chalk'`
- `import './helpers.js'` → `import './helpers.js'` resolved to `http://localhost:5710/preview/workspace/skills/my-skill/helpers.js`

The rewritten source is bundled into a `Blob` and executed via `await import(blobUrl)`. The blob's internal imports use absolute URLs — they go straight to the network, no SW interception needed. The preview SW still serves `/__shims/*` as planned (called by absolute URL).

### Local File Imports

Local relative imports resolve to absolute preview SW URLs during the rewrite step. The preview SW (scoped to `/preview/`) serves these correctly since they are direct network requests (not requests from the main-page context). Mounted directories are supported via the existing SW fallback to the main-page VFS bridge.

### CLI vs Extension Mode

- **CLI mode:** Specifier rewriting + blob URL `import()` in the main document.
- **Extension mode:** Same rewriting approach — rewritten code sent via `esm_exec` postMessage to `sandbox.html` which executes it as a blob URL (sandbox is CSP-exempt, blob imports work).

## What Doesn't Change

- The entire CJS/require path — untouched
- Preview SW's existing `/preview/*` VFS serving — untouched
- Tool definitions, shell registration — untouched

## Known Limitations

### Private / scoped npm packages

The esm.sh CDN only has access to the public npm registry. Private or scoped packages hosted on private registries will fail to load. This also affects packages that are public but have complex build steps or native dependencies that esm.sh cannot handle.

Possible future solutions:

- Vendor the package locally into the skill folder and import via relative paths
- Serve from VFS if the package exists in a local `node_modules/` on VFS
- Configurable registry to point scoped packages to a custom registry URL

### npm package binaries

Packages that provide CLI binaries (via the `bin` field in `package.json`) are not supported. esm.sh serves JavaScript modules, not executable scripts. Skills that need to invoke package binaries cannot do so through this import mechanism.

Possible future solutions:

- Shell-level `npx`-like command that fetches and runs package binaries
- Pre-installed binaries bundled into the VFS
