# CLAUDE.md

This file covers the repo's developer-tooling surface.

## Scope

`packages/dev-tools/` is the home for build helpers, QA setup guidance, and developer verification utilities. Some of that tooling still lives at the repo root while the modularization settles; treat the locations below as the active surface.

## Key Tooling Areas

- **Prompt/build helpers**: `packages/dev-tools/tools/slicc-prompt.mjs`
- **Build configs**: `packages/webapp/vite.config.ts`, `packages/chrome-extension/vite.config.ts`, `biome.json`
- **QA setup**: `packages/node-server/src/qa-setup.ts` plus the root `npm run qa:*` scripts
- **Visual/integration helpers**: `tests/test-dips.mjs` and related targeted test utilities
- **RUM error triage** (error-to-insight pipeline): `packages/dev-tools/rum-error-triage/` — run `node packages/dev-tools/rum-error-triage/triage-rum-errors.mjs` to query RUM for new SLICC errors and write triage candidates; pure logic in `lib.mjs` (tested via the `dev-tools` vitest project). Driven nightly by `.github/workflows/rum-error-triage.yml`. See its `README.md`.
- **Doc size check** (`npm run lint:docs`): `packages/dev-tools/tools/check-doc-sizes.mjs` — enforces size budgets for the repo's machine-read instruction files: root `CLAUDE.md` ≤ 30000 chars, `packages/vfs-root/shared/CLAUDE.md` ≤ 3000 bytes, and every GitHub Copilot instruction file (`.github/copilot-instructions.md` plus each `.github/instructions/*.instructions.md`, discovered dynamically) ≤ 4000 chars (Copilot's truncation limit). Non-zero exit on violation. Also runs in the `.husky/pre-commit` hook (once, when a budgeted file is staged, after `lint-staged`).
- **Linear-history check**: `packages/dev-tools/tools/check-linear-history.sh` — fails if a PR branch contains merge commits in `base..head` (i.e. `main` was merged in instead of rebased). Run `bash packages/dev-tools/tools/check-linear-history.sh [base-ref] [head-ref]` (defaults base to `origin/${GITHUB_BASE_REF:-main}`, falling back to `${GITHUB_BASE_REF:-main}` when that remote-tracking ref isn't present locally; head to `HEAD`). Driven by the `linear-history` job in `.github/workflows/ci.yml`, which catches merge-queue linear-history rejections at PR time instead of after a green run.
- **Skill lint** (`npm run lint:skills`): `packages/dev-tools/tools/lint-skills.mjs` — runs tessl skill import + lint over all 12 `SKILL.md` skills via the `@tessl/cli` npm path (ephemeral plugin metadata in a temp copy, never committed). Warns and skips (exit 0) when tessl is unresolvable; pass `--strict` (CI) to fail instead.
- **Patch reconcile** (`npm run lint:patches`): `packages/dev-tools/patch-reconcile/` — keeps version-pinned `patch-package` patches (`patches/`) honest across Renovate bumps. `check-patches.mjs` (the guard, chained into `lint` / `lint:ci`) fails on an undocumented, out-of-sync, or **orphaned** patch (filename version ≠ `package-lock.json` version) by reading only the lockfile + `patches/patches.json` (no install). `reconcile-context.mjs` emits the orphaned patches + manifest metadata (markdown / `--json`) for `.github/workflows/renovate-patch-reconcile.yml`, which hands patched-dependency Renovate PRs to `claude-code-action` to regenerate-or-remove the patch. Pure logic in `lib.mjs` (vitest project `dev-tools`). See `patches/README.md`.
- **innerHTML guard** (`npm run lint:no-innerhtml`): `packages/dev-tools/tools/check-no-innerhtml.mjs` — enforces the `@slicc/webcomponents` "no innerHTML" rule. Fails on any `.innerHTML =` / `.outerHTML =` assignment or `insertAdjacentHTML(...)` call in shipped component source (`packages/webcomponents/src/**/*.ts`), so the library's DOM-construction decision (build via `internal/dom.ts` `h()`/`sheet()` + `iconEl()`, never an HTML string) is self-enforcing. Comments/JSDoc and read comparisons (`=== `) don't trip it; `*.stories.ts` (Storybook scaffolding, excluded from `dist/`) and `*.test.ts` are exempt. Chained into both `npm run lint` and `npm run lint:ci` (so the CI lint gate covers it).
- **Storybook PR screenshots** (`@slicc/webcomponents` visual spot-check): `packages/dev-tools/tools/storybook-affected-screenshots.mjs` (capture driver) + `packages/dev-tools/tools/storybook-affected-stories-lib.mjs` (pure affected-story resolver, tested via the `dev-tools` vitest project). Local run: `npm run build-storybook -w @slicc/webcomponents && node packages/dev-tools/tools/storybook-affected-screenshots.mjs --changed-files=<path> --storybook-static=packages/webcomponents/storybook-static --out=<dir>` (requires `npx playwright install chromium` once). Writes `<out>/manifest.json` (flat `shots[]` schema, v1). Driven on PR by `.github/workflows/storybook-screenshots.yml`; see [`packages/webcomponents/CLAUDE.md`](../webcomponents/CLAUDE.md) for the end-to-end flow.
- **Dead code detection — production-mode files** (`npm run deadcode:production-files`): `knip --production --include files` (selector flags pinned in `package.json` to keep the CI gate from drifting from the documented local command, matching the existing `deadcode*` convention). Runs in the `lint` job of `.github/workflows/ci.yml` right after the default `deadcode` step. Closes the gap that the default knip gate misses: with test files as entry points, a module imported only by a test (and otherwise wired to nothing) looks reachable; `--production` drops test entries and surfaces those test-only dead files (precedent: `api-key-dialog.ts`, issue #1018). Files-only by design — individual unused exports remain intentional API surface and are not enforced here. **Production-suffix discipline**: every workspace `entry`/`project` glob that's part of the production graph MUST be `!`-suffixed (`src/**/*.ts!`, `vite.config.ts!`); knip's `--production` flag keeps only suffixed patterns, so an unsuffixed entry would silently turn the workspace inert under the gate. Legitimate test-only fixtures (currently the two cloud-core fixtures `tests/fixtures/index.ts` and `tests/fixtures/mem-registry.ts`, both reachable only from `*.test.ts` files) are excluded via negated `project` patterns (`!tests/fixtures/index.ts!`) in the `packages/cloud-core` workspace in `knip.json` — `ignoreFiles` with the `!` suffix does NOT work (the suffix is passed straight to picomatch), so negated production-graph patterns are the only mechanism that both keeps fixtures out of the dead-files set AND avoids the `Remove unused ignore` hint in the default gate.
- **Function-size + cognitive-complexity debt boy-scout gate**: `packages/dev-tools/tools/check-touched-exemptions.mjs` (+ pure lib `size-exemption-lib.mjs`, vitest project `dev-tools`). One driver enforces BOTH per-rule debt blocks in `biome.json` in a single pass: the `overrides` block whose ONLY rule is `complexity.noExcessiveLinesPerFunction: "off"` (function-size debt list) AND the parallel `overrides` block whose ONLY rule is `complexity.noExcessiveCognitiveComplexity: "off"` (cognitive-complexity debt list). Exits non-zero if any file changed in the PR is still on EITHER list, naming each offending file and which rule it violates with a rule-appropriate fix-it message. The gate ALSO blocks growing either debt list: it compares the current `biome.json` against the BASE ref's `biome.json` (`git show <base>:biome.json`) and fails on any glob added to either list — except when the list is being introduced (base had no entries for that rule), in which case bootstrapping additions are allowed. If the base config can't be read (e.g. local dev without a fetched base), the added-entry check is skipped with a notice and the touched-files check still runs. CI step `Function-size + cognitive-complexity debt boy-scout gate` runs it via `node packages/dev-tools/tools/check-touched-exemptions.mjs` (`fetch-depth: 0` on the lint checkout so merge-base resolves against `$GITHUB_BASE_REF`); auto-skips on `merge_group` / `push` events. Local run: `node packages/dev-tools/tools/check-touched-exemptions.mjs [base-ref]` (defaults to `origin/main`), or override the input with `CHANGED_FILES=path1,path2 …`.

- **SLICC CDP debug toolkit**: `packages/dev-tools/tools/slicc-debug.mjs` — CDP-based diagnostic CLI for the standalone dev harness. Connects to Chrome's remote debugging port to inspect a running SLICC instance. Subcommands: `targets` (list CDP targets), `logs [--target=page|worker]` (stream console output/errors), `vfs ls <path>` / `vfs cat <path>` (explore the VFS via BroadcastChannel), `eval <expr>` / `eval --target=worker <expr>` (evaluate JS in page or kernel worker), `shell <command>` (run a shell command in the SLICC terminal via the sprinkle exec bridge), `chat <prompt>` (send a prompt to the agent). Set `SLICC_CDP_PORT` env var or let it auto-detect. Run: `node packages/dev-tools/tools/slicc-debug.mjs --help`.

- **Swift-server fresh bridge harness**: `packages/dev-tools/tools/dev-swift-fresh.sh` — brings up the native macOS `swift-server` thin-bridge against the hosted webapp on a fresh, isolated Chrome profile so it can run alongside a live node-server harness without port/profile collisions. Builds `slicc-server`, resolves a Chrome for Testing binary, and launches the bridge on `:5720` (Chrome CDP `:9224`) pointed at the local wrangler UI on `:8787` (reused if already running). Sets `BRIDGE_DEV_ALLOWED_ORIGINS=http://localhost:8787` so the cross-origin leader tab is allowlisted. Run: `bash packages/dev-tools/tools/dev-swift-fresh.sh`. Pairs with `slicc-debug.mjs` (`SLICC_CDP_PORT=9224 SLICC_BRIDGE_PORT=5720`) for verification.

## What Lives Here Conceptually

- scripts that support local development rather than runtime behavior
- config files that shape builds or verification flows
- QA setup flows for isolated profiles and tray testing
- one-off utilities used by release, validation, or inspection workflows

## Usage Notes

- Prefer root npm scripts when a helper already has one.
- Keep dev-only configs and utilities out of runtime packages unless they are required at runtime.
- When adding new tooling, document both the file location and the intended entry command.
